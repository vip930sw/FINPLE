# FINPLE 바이브코딩 경쟁력·방어력 개선 로드맵

작성일: 2026-06-26

## 1. 문서 목적

이 문서는 FINPLE이 생성형 AI와 코드 에이전트를 활용해 빠르게 제품을 구현한 상황에서, 단순한 화면·기능 복제를 넘어 지속 가능한 핀테크 SaaS 경쟁력을 확보하기 위한 개선 방향을 정리한다.

핵심 전제는 다음과 같다.

> 바이브코딩은 개발수단이며, 그 자체가 사업의 약점도 경쟁우위도 아니다. FINPLE의 장기 경쟁력은 데이터 품질, 재현 가능한 계산, 운영 안정성, 사용자 신뢰, 고객 데이터, 콘텐츠와 제휴에서 만들어야 한다.

## 2. 현재 판단

### 2.1 바이브코딩이 낮추는 장벽

생성형 AI는 다음 영역의 진입장벽을 크게 낮춘다.

- 홈페이지와 대시보드 UI
- 자산 입력·비중 설정
- 기본 복리·적립식 계산
- 일반적인 CAGR·MDD·BETA 표시
- 로그인·저장·결제의 표준 구현
- 정형화된 리포트 화면

따라서 작동하는 MVP가 있다는 사실만으로는 과거보다 높은 기술적 희소성을 인정받기 어렵다.

### 2.2 FINPLE에서 아직 복제하기 쉬운 부분

- 화면 구성과 브랜드 톤
- 포트폴리오 입력·편집 UX
- 공개된 공식에 따른 기본 지표 계산
- 고정 문구 기반 상세분석
- 표준형 회원가입·구독·MY PAGE

### 2.3 FINPLE에서 방어력으로 전환할 수 있는 부분

- 한국·미국 자산 데이터 정제·검수 이력
- Raw·Rolling CAGR, MDD, BETA, 배당 산정정책
- 데이터 이상치 reason code와 수동 검수 라벨
- 재현 가능한 포트폴리오 계산 엔진
- 금융서비스용 출력 검증과 금지표현 차단
- 사용자 행동·재방문·저장·결제 데이터
- 금융교육 콘텐츠, 기관 PoC, 데이터·핀테크 제휴

## 3. 현재 방어력 임시 평가

아래 평가는 법적·회계적 기업가치 평가가 아니라 제품 전략을 위한 내부 상대평가이다.

| 평가대상 | 현재 방어력 | 설명 |
| --- | ---: | --- |
| 화면과 UI | 2/10 | 유사 화면과 흐름을 AI로 비교적 쉽게 구현 가능 |
| 기본 계산·시뮬레이션 기능 | 3/10 | 공식은 공개돼 있고 일반적인 웹개발 범위에 해당 |
| 데이터·인증·결제·운영 포함 전체 제품 | 4/10 | 복제 가능하지만 실제 안정화와 검수에 시간이 필요 |
| 데이터 품질·사용자·제휴 축적 후 목표 | 6~7/10 | 검수 라벨, 운영이력, 고객 신뢰, 유통망이 결합될 경우 |

## 4. 개선 원칙

### 원칙 1. 코드가 아니라 검증 가능한 결과를 자산화한다

- 계산식 버전
- 입력 데이터 기준일
- 데이터 출처
- 보정 여부와 보정 사유
- 품질점수
- 검수 상태
- 수정 전후 값
- 테스트 결과

### 원칙 2. AI와 결정론적 계산의 책임을 분리한다

- CAGR·MDD·평가금액·배당 등 숫자는 검증 가능한 계산 엔진이 확정한다.
- AI는 확정된 숫자를 해석하고 설명한다.
- AI가 숫자를 새로 생성하거나 매수·매도 판단을 제공하지 않도록 한다.

### 원칙 3. 기능 수보다 반복 사용과 신뢰를 우선한다

- 사용자가 어디에서 이탈하는지 측정한다.
- 저장·재방문·비교·PDF 등 핵심 행동을 추적한다.
- 설명이 실제 이해도를 높였는지 검증한다.
- 신규 기능은 사용자 행동 개선 여부로 평가한다.

### 원칙 4. 패치 누적보다 유지보수 가능한 구조로 전환한다

- DOM 후처리 패치와 import 순서 의존성을 점진적으로 제거한다.
- 핵심 기능은 React 컴포넌트와 서비스 계층으로 환원한다.
- 기능별 회귀테스트를 갖춘 뒤 리팩터링한다.
- 한 번에 대규모 재작성하지 않고 화면·기능 단위로 교체한다.

## 5. 우선순위별 실행계획

## P0. 즉시 착수: 신뢰성과 재현성 확보

목표 기간: 0~4주

### P0-1. 지표 산정정책과 데이터 계보 확정

실행 항목:

- 자산별 데이터 기준일·기간·출처 저장
- Raw CAGR과 Rolling CAGR의 역할 분리
- 가격수익과 총수익 분리
- MDD 직접계산과 포트폴리오 근사값 분리
- BETA 기준 벤치마크 명시
- 환율·통화·proxy 사용 여부 명시
- 데이터 상태값을 `ready`, `warning`, `review`, `invalid` 등으로 통일

완료 기준:

- 동일 입력으로 동일 산출물이 재현된다.
- 값이 바뀐 경우 원천 데이터와 정책 버전을 추적할 수 있다.

### P0-2. Golden Portfolio 회귀테스트 구축

대표 포트폴리오를 고정하고 코드 변경마다 자동 비교한다.

예시:

- QQQ 100%
- QQQ·SCHD 50:50
- 주식·채권 60:40
- QQQ·SCHD·TLT·GLD 혼합
- 한국 ETF 혼합
- 일시금과 월 적립식
- 배당 재투자 적용·미적용
- 환율 적용·미적용

완료 기준:

- 핵심 계산 로직 변경 시 기대값 차이를 자동 탐지한다.
- 허용오차와 변경승인 절차가 문서화된다.

### P0-3. 기술부채와 패치 의존성 감사

실행 항목:

- `main.jsx` import 순서 의존성 목록 작성
- `*Patch.js`별 대상 화면·DOM·이벤트 정리
- MutationObserver·innerHTML·전역 이벤트 사용처 분류
- 제거·유지·컴포넌트 환원 대상을 우선순위화
- MY PAGE, 결제, 시뮬레이터부터 안전하게 정리

완료 기준:

- 핵심 화면별 단일 책임 파일과 회귀테스트가 있다.
- 신규 기능은 임시 DOM 패치보다 정식 컴포넌트 구현을 우선한다.

### P0-4. 보안·운영 기본선 점검

실행 항목:

- Secret 노출 검사
- 의존성 취약점 점검
- 인증·결제·웹훅 회귀테스트
- DB 백업과 복구절차
- 관리자 작업이력과 오류로그
- 개인정보 최소수집·보관기간 점검
- 장애·문의·환불 대응절차 정리

완료 기준:

- 출시 전·배포 후 체크리스트가 자동화 또는 반복 가능하게 정리된다.

## P1. 1~3개월: 데이터 품질을 제품 경쟁력으로 전환

### P1-1. Data Sentinel 운영화

실행 항목:

- reason code와 severity를 운영 데이터에 연결
- 수동 검수 결과를 라벨 데이터로 축적
- 자산별 품질점수와 검수상태 제공
- 오류·보정·확정 이력 저장
- 규칙형 결과와 ML 이상치 결과를 분리 표시

핵심 KPI:

- 수동 검수시간 감소율
- 잘못된 경보 비율
- 중요 오류 탐지율
- 검수 대기 자산 수
- 데이터 갱신 소요시간

### P1-2. 재현 가능한 지표 엔진 구축

실행 항목:

- 월간 시계열 데이터 감사
- CAGR·MDD·BETA 재산출 스크립트
- 신규 overlay 자동 생성
- 산출 정책 버전 기록
- 대표 ETF·지수 peer 비교
- 시계열 직접 포트폴리오 MDD와 회복기간 계산

### P1-3. 데이터 품질 표시를 사용자 가치로 전환

사용자 화면에서 내부 용어를 그대로 노출하지 않고 다음 정보를 이해하기 쉽게 제공한다.

- 데이터 기준일
- 분석기간
- 산출방식
- 보정 여부
- 신뢰도 또는 데이터 상태
- 주의가 필요한 이유

## P1. 1~3개월: 고객 검증과 유통 기반 구축

### P1-4. 첫 고객군을 좁힌다

권장 초기 타깃:

> 미국 ETF를 2개 이상 보유하고 월 적립식 투자 중인 30~40대 직장인

검증할 문제:

- 전체 포트폴리오의 장기 성과·위험·실질가치를 한 화면에서 비교하기 어렵다.
- 자산을 추가하거나 비중을 바꿨을 때 결과 변화가 직관적이지 않다.
- CAGR·MDD·BETA를 실제 의사결정에 어떻게 활용할지 어렵다.

### P1-5. 핵심 퍼널을 측정한다

측정 항목:

- 첫 방문 → 시뮬레이터 진입
- 자산 입력 완료
- 상세분석 확인
- 저장
- 7일 재방문
- PDF·비교 사용
- 결제수단 등록
- Personal 전환

초기 목표 예시:

- 누적회원 50명 이상
- 실제 사용테스트 20명 이상
- 완주형 설문 15건 이상
- 저장 50건 이상
- 7일 내 재방문 사용자 10명 이상
- 실제 유료고객 3~5명 이상

## P2. 3~6개월: 차별화 기능과 신뢰자산 구축

### P2-1. Asset DNA

- 자산 역할 자동 분류
- 가격·위험 특성 기반 유사자산 탐색
- 단순 태그가 아닌 데이터 기반 분류확률 제공
- 특정 종목 추천이 아니라 자산 특성 이해를 지원

### P2-2. AI 분석 고도화

- 검증된 지표만 입력
- 숫자 환각 차단
- 금지문구·종목권유 차단
- 데이터 한계와 분석조건 자동 고지
- 사용자 이해도 평가와 설명문 A/B 테스트

### P2-3. 시나리오 분석

- Rolling historical baseline
- Block Bootstrap
- P10·P50·P90 확률밴드
- 하락확률과 손실규모
- 회복기간
- 시장 벤치마크 비교
- BETA 기반 스트레스 테스트

### P2-4. 콘텐츠·브랜드·제휴

- CAGR·MDD·실질가치 교육 콘텐츠
- 통제된 포트폴리오 비교 실험
- 직장인·퇴직연금 교육 프로그램
- 금융교육기관 PoC
- 핀테크·데이터 기업 협력
- 기관용 리포트 또는 API 가능성 검토

## P3. 6~12개월: 복제하기 어려운 축적자산 확대

- 검수 라벨 데이터셋
- 실제 사용자가 어려워하는 지표·설명 데이터
- 포트폴리오 유형별 이탈·재방문 패턴
- 데이터 공급·교육·기관 제휴
- 내부 계산·검수 프로세스의 지식재산권 검토
- 기관용 데이터 품질 리포트와 API
- 운영감사·보안·소비자보호 체계

## 7. 외부 설명 권장문구

> FINPLE은 생성형 AI와 코드 에이전트를 활용해 개발 생산성을 높였습니다. 다만 핵심 경쟁력은 코드 생성 자체가 아니라 한국·미국 금융데이터의 정제·검수 체계, 재현 가능한 포트폴리오 계산 엔진, 금융서비스 운영·규제 기준, 그리고 사용자 사용 데이터를 통해 지속적으로 개선되는 분석 품질에 있습니다.

피해야 할 표현:

- AI가 전부 만든 서비스
- 누구나 쉽게 만든 앱
- 종목을 추천하는 AI
- 수익률을 예측하는 AI
- 자동으로 최적 포트폴리오를 제시
