# FINPLE 업데이트 페이지 운영 및 인수인계 가이드

- 공개 페이지: `https://finple.co.kr/updates`
- 업데이트 데이터: `src/data/updatesData.js`
- 화면 구성: `src/components/UpdatesPage.jsx`
- 화면 스타일: `src/App.css`
- 기준 시간대: `Asia/Seoul`

## 1. 공개 페이지의 목적

`/updates`는 FINPLE 사용자가 새 기능, 기능 개선, 오류 수정과 데이터 변경을 쉽게 확인하는 페이지다.

공개 페이지에는 개발 과정의 용어를 직접 노출하지 않는다. 작업 번호, 코드 변경 식별자, 내부 단계명과 같은 추적 정보는 GitHub 기록에만 남기고, 페이지에는 사용자가 실제로 달라진 점을 설명한다.

## 2. 작성 원칙

1. 실제 반영일을 기준으로 날짜를 기록한다.
2. 각 카드에 `Beta v0.0.0` 형식의 버전을 표시한다.
3. 제목은 결과를 한 문장으로 설명한다.
4. 요약은 사용자가 체감하는 변화와 목적을 설명한다.
5. 세부 내용은 `새 기능`, `개선`, `수정`, `데이터`, `안내` 중 필요한 항목만 사용한다.
6. 같은 날 또는 가까운 시기에 진행된 비슷한 작업은 한 카드로 묶을 수 있다.
7. 작업이 있었던 기간을 지나치게 큰 날짜 범위 하나로 합치지 않는다. 중요한 변경일을 빠뜨리지 않도록 확인한다.
8. 모든 카드는 처음에는 접힌 상태로 표시하고, 사용자가 원하는 카드만 펼쳐 확인하도록 한다.

## 3. 데이터 작성 방법

`src/data/updatesData.js`의 `UPDATES` 배열 맨 위에 최신 항목을 추가한다.

```js
{
  id: "beta-v0-4-7",
  version: "Beta v0.4.7",
  date: "2026-06-21",
  title: "사용자가 확인할 수 있는 변경 결과",
  summary: "무엇이 달라졌고 왜 편리해졌는지 설명합니다.",
  categories: ["new", "improved"],
  changes: {
    added: ["새로 제공하는 기능"],
    improved: ["기존보다 편리해진 점"],
    fixed: ["해결한 오류"],
  },
}
```

- `id`: 페이지 안에서 중복되지 않는 영문 식별자
- `version`: 공개할 Beta 버전
- `date`: `YYYY-MM-DD`
- `categories`: 상단 필터에 사용할 분류
- `changes`: 실제로 존재하는 항목만 작성

후속 작업은 같은 파일의 `ROADMAP_ITEMS`에서 관리한다. 이미 완료된 항목은 후속 작업에서 제거하고 해당 업데이트 카드에 반영한다.

## 4. 공개 문구와 내부 기록의 경계

공개 페이지에는 다음 내용을 쓰지 않는다.

- 작업 번호와 코드 변경 식별자
- 내부 파일명, 함수명, 배포 명령
- 개발자만 이해할 수 있는 오류 메시지
- `작업일지`, `커밋`, `PR`, `Step` 같은 구현 중심 표현

내부 추적은 GitHub 변경 요청의 설명과 관련 인수인계 문서에 남긴다. 공개 카드의 날짜·버전·내용이 어느 변경 요청에서 수정됐는지 GitHub 설명에서 연결하면 된다.

## 5. 화면 유지 기준

- 현재의 흰색 카드와 파란색 강조 색상을 유지한다.
- 업데이트 목록, 후속 작업 상자의 좌우 폭을 동일하게 유지한다.
- 데스크톱과 모바일 모두 카드 전체가 눌러지는 접기·펼치기 방식으로 제공한다.
- 카드 안에 다시 접는 영역을 만들지 않는다.
- 버튼은 키보드로 접근할 수 있어야 하며 펼침 상태를 보조기기에 전달한다.
- 마지막 상자와 푸터가 붙지 않도록 데스크톱 96px, 모바일 72px 수준의 하단 여백을 유지한다.

## 6. 변경 절차

```powershell
git fetch origin
git switch main
git merge --ff-only origin/main
npm.cmd run build
npx.cmd eslint src/components/UpdatesPage.jsx src/data/updatesData.js
git diff --check
```

1. 최신 `main`을 받는다.
2. GitHub 변경 이력과 실제 서비스 반영일을 확인한다.
3. 비슷한 변경을 사용자 관점에서 묶고 날짜와 Beta 버전을 정한다.
4. `updatesData.js`를 수정한다.
5. 완료된 후속 작업을 정리한다.
6. 빌드, 코드 검사와 화면 검증을 진행한다.
7. 변경 요청 설명에 추가·수정한 날짜와 버전을 기록한다.

## 7. 검수 체크리스트

- [ ] 최신 항목이 배열과 화면의 가장 위에 있는가
- [ ] 실제 작업이 있었던 주요 날짜가 누락되지 않았는가
- [ ] 모든 항목에 Beta 버전이 표시되는가
- [ ] 개발자용 용어가 공개 문구에 남지 않았는가
- [ ] 카드가 모두 접힌 상태로 시작하는가
- [ ] 필터 선택 후에도 해당 카드가 정상적으로 펼쳐지는가
- [ ] 업데이트 목록과 후속 작업 상자의 폭이 같은가
- [ ] 마지막 상자와 푸터 사이에 충분한 여백이 있는가
- [ ] 모바일에서 긴 제목과 필터가 잘리지 않는가
- [ ] 완료된 항목이 후속 작업에 중복되어 있지 않은가

## 8. 이번 개편의 인수인계 사항

- 이전의 자동 생성 방식은 개발 기록이 공개 화면에 그대로 드러나는 문제가 있어 제거했다.
- 공개 데이터는 `updatesData.js`에서 사람이 검토한 사용자용 문장으로 관리한다.
- 2026년 5월 10일부터 6월 20일까지의 기록은 실제 변경일을 기준으로 나누되, 같은 목적의 작업은 하나의 카드로 묶었다.
- 버전은 서비스가 정식 출시 전임을 알 수 있도록 모두 `Beta`로 표기한다.
- 내부 작업 추적은 GitHub에서 유지하고, 공개 페이지는 사용자 안내에 집중한다.
