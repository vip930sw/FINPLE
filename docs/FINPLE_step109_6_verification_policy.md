# FINPLE Step 109-6 — 인증 방식 검토 및 회원가입 입력 기준

## 1. 결론

FINPLE의 현재 단계에서는 **이메일 인증을 우선 적용**한다.

휴대전화 본인확인은 결제 심사, 고액 결제, 부정 이용 방지, 본인확인 의무가 명확해지는 단계에서 별도로 검토한다.

```text
1차 적용: 이메일 인증
2차 확장: Google / Kakao / Naver OAuth
추후 검토: 휴대전화 본인확인
비권장: 주민등록번호 직접 수집
```

## 2. 이메일 인증이 우선인 이유

```text
장점
- 구현 비용과 운영 비용이 낮다.
- 회원가입, 비밀번호 재설정, 알림 발송과 자연스럽게 연결된다.
- 수집 개인정보가 상대적으로 적다.
- SaaS형 웹서비스의 기본 계정 식별 수단으로 적합하다.

단점
- 실명 확인은 되지 않는다.
- 1인이 여러 이메일로 가입할 수 있다.
- 인증 메일이 스팸함으로 들어갈 수 있다.
```

FINPLE은 현재 투자 분석 도구이며, 직접 금융거래를 중개하거나 실명 기반 금융계좌를 개설하는 서비스가 아니다. 따라서 초기 회원가입 단계에서는 이메일 인증으로 충분하다.

## 3. 휴대전화 인증의 위치

```text
장점
- 실사용자 여부 확인에 강하다.
- 중복가입, 악성가입, 부정 결제 방지에 유리하다.
- 국내 사용자는 익숙하다.

단점
- PASS / 본인확인기관 / 문자 인증 등 외부 사업자 연동이 필요하다.
- 비용과 운영 부담이 생긴다.
- 휴대전화번호는 개인정보 민감도가 높아진다.
- 서비스 진입장벽이 높아질 수 있다.
```

따라서 FINPLE에서는 휴대전화 인증을 기본 회원가입 절차로 넣지 않고, 아래 상황에서만 검토한다.

```text
- 결제수단 등록
- 구독 결제 이상 탐지
- 계정 복구
- 고위험 부정 사용 방지
- 사업자/전문가 계정 검증
```

## 4. 주민등록번호 수집 기준

주민등록번호는 FINPLE이 직접 수집하지 않는 것을 원칙으로 한다.

필요 시에도 FINPLE 서버에 주민등록번호를 저장하지 않고, 본인확인기관 또는 결제사가 처리한 결과값만 받는 구조가 안전하다.

```text
수집하지 않음:
- 주민등록번호 원문
- 카드번호 원문
- 계좌번호 원문

필요 시 보관 가능한 값:
- 결제사 customerKey
- billingKey 식별값
- 본인확인 완료 여부
- 본인확인 일시
- 구독 상태
- 결제 이력
```

## 5. 회원가입 화면 입력 기준

회원가입 화면에서는 이름과 닉네임/ID를 분리한다.

```text
이름
- 사용자 본인이 식별할 수 있는 이름
- 추후 결제/문의/계정 관리에서 사용 가능

닉네임 / ID
- 서비스 화면에서 표시할 별칭
- 커뮤니티 기능이나 공개 표시가 생길 경우 확장 가능
```

현재 프론트 회원가입 화면은 아래 입력을 받는다.

```text
1. 이름
2. 닉네임 / ID
3. 이메일
4. 비밀번호
5. 비밀번호 확인
6. 이용약관 동의
7. 개인정보처리방침 동의
8. 마케팅 안내 수신 동의
```

## 6. 이메일 인증 도입 예정 흐름

```text
회원가입 입력
→ 이메일 중복확인
→ 회원가입 요청
→ 인증 메일 발송
→ 인증 전 상태: 이메일 인증 필요
→ 인증 링크 클릭
→ /verify-email?token=...
→ 인증 완료
→ MY PAGE / 서버 저장 활성화
```

## 7. 단계별 적용 제안

```text
Step 109-6A
- 회원가입 화면에서 이름과 닉네임/ID 입력 분리
- 인증 정책 문서화

Step 109-6B
- auth_email_tokens 사용 여부 점검
- 이메일 인증 토큰 생성/검증 API 추가

Step 109-6C
- 메일 발송 서비스 선택
- Resend 또는 SendGrid 우선 검토

Step 109-6D
- /verify-email 화면 추가
- 인증 전 MY PAGE / 서버 저장 제한

Step 109-7
- Google OAuth 검토

Step 109-8
- Kakao OAuth 검토

Step 109-9
- Naver OAuth 검토
```
