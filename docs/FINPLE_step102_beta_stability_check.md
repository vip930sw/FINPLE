# FINPLE Step 102. 베타 운영 안정성 점검

## 1. 목적

Step 102는 기능 추가가 아니라, 베타 운영 중 사용자가 실제로 겪을 수 있는 접속 지연, 저장 실패, 조회 제한, 문의 누락, 관리자 접근 문제를 사전에 점검하는 단계입니다.

현재 FINPLE은 소수 사용자 대상 베타 공개 가능 상태이며, 이번 단계의 목표는 다음과 같습니다.

- 서비스 접속 안정성 확인
- 프론트/백엔드 연결 안정성 확인
- 포트폴리오 서버 저장/불러오기 안정성 확인
- Alpha Vantage 조회 제한 상황에서 사용자 안내가 충분한지 확인
- 문의사항 접수와 관리자 조회가 안정적으로 동작하는지 확인
- 베타 운영 중 장애 발생 시 우선순위와 대응 기준 정리

---

## 2. 점검 대상

| 구분 | 점검 대상 | 기준 |
|---|---|---|
| 프론트 | Vercel 배포, 도메인, 라우팅 | 직접 접속 가능 |
| 백엔드 | Render Web Service, /api/health | 정상 응답 |
| DB | Supabase 저장/조회 | 포트폴리오·문의 저장 정상 |
| 외부 API | Alpha Vantage | 조회 제한 시 캐시/수동값 유지 |
| 관리자 | /admin, 관리자 토큰 | 토큰 없이는 관리자 조회 불가 |
| 사용자 UX | 실패 안내 문구 | 사용자가 다음 행동을 이해 가능 |

## 3. 매일 1회 운영 점검

베타 공개 후 첫 1주일은 하루에 한 번 아래만 확인합니다.

### 3-1. 접속 확인

- [ ] https://finple.co.kr 접속 가능
- [ ] 홈 화면 정상 표시
- [ ] 로그인 화면 정상 표시
- [ ] 요금제 / 문의사항 / 법적 문서 이동 정상
- [ ] https://finple.co.kr/admin 직접 접속 가능
- [ ] 홈으로 이동 시 주소가 https://finple.co.kr/ 로 정리됨

### 3-2. 백엔드 Health Check

브라우저 또는 API 테스트 도구에서 확인합니다.

- [ ] https://finple-api.onrender.com/api/health 정상 응답
- [ ] 응답에 ok: true 표시
- [ ] 첫 요청이 느려도 이후 요청이 정상 응답
- [ ] Render 로그에 반복 오류가 없음

확인 메모:

```text
확인일:
응답 시간:
오류 여부:
특이사항:
```

### 3-3. 포트폴리오 저장 / 불러오기

- [ ] 체험 계정으로 시작
- [ ] Free 플랜에서 포트폴리오 1개만 표시
- [ ] 자산 수량 또는 현재가 변경
- [ ] 서버 저장 성공
- [ ] 새로고침 후 서버 불러오기 성공
- [ ] 저장 시점 값이 복구됨
- [ ] 포트폴리오가 중복 생성되지 않음

### 3-4. 문의사항

- [ ] 문의사항 작성 가능
- [ ] 답변 이메일 없이도 접수 가능
- [ ] 접수 완료 문구 표시
- [ ] /admin에서 문의 목록 조회 가능
- [ ] 상태 변경 가능
- [ ] 종료 처리 가능

## 4. Render Free 인스턴스 지연 점검

현재 백엔드는 Free 인스턴스를 사용하므로, 일정 시간 사용하지 않으면 첫 요청이 느릴 수 있습니다.

### 확인 방법

1. 30분 이상 기다린 뒤 사이트 접속
2. 체험 계정 로그인 또는 서버 불러오기 실행
3. 첫 요청 지연 시간을 체감 기록
4. 두 번째 요청은 정상 속도로 돌아오는지 확인

### 판단 기준

| 상태 | 판단 |
|---|---|
| 첫 요청 10~60초 지연, 이후 정상 | Free 인스턴스 특성으로 허용 |
| 매번 60초 이상 지연 | P1 개선 검토 |
| 저장/불러오기 실패 | P0 또는 P1 |
| 사용자가 실패로 오해할 가능성 높음 | 안내 문구 개선 |

### 사용자 안내 후보

```text
서버가 잠시 대기 상태일 수 있습니다. 첫 요청은 수십 초 정도 걸릴 수 있으며, 잠시 후 다시 시도해 주세요.
```

## 5. Alpha Vantage 조회 제한 점검

현재 FINPLE은 API 호출을 절약하기 위해 최근 조회값과 캐시값을 활용합니다.

### 확인할 것

- [ ] 전체 조회 버튼 동작
- [ ] 개별 조회 버튼 동작
- [ ] 조회 제한 시 기존 수동값/캐시값 유지
- [ ] 조회 제한 안내 문구가 사용자에게 이해되는지
- [ ] CAGR / BETA / MDD / 배당률은 API 안정 제공값이 아닐 수 있음을 안내

### 판단 기준

| 상태 | 판단 |
|---|---|
| 가격 일부 조회 실패, 기존값 유지 | 허용 |
| 모든 가격 조회 실패 | P1 |
| 오류 메시지가 이해 불가 | P2 |
| 사용자가 실제 투자 데이터로 오해 | P1 문구 개선 |

## 6. 서버 저장 실패 대응 기준

서버 저장/불러오기는 베타 운영에서 가장 중요한 기능입니다.

### 주요 실패 가능성

- Render cold start
- CORS 설정 누락
- Supabase 연결 지연
- 사용자 ID 누락
- 네트워크 실패
- 중복 포트폴리오 생성

### 우선순위

| 오류 | 우선순위 |
|---|---|
| 저장이 전혀 안 됨 | P0 |
| 불러오기가 전혀 안 됨 | P0 |
| 새로고침 후 데이터 유실 | P0 |
| 중복 포트폴리오 생성 | P1 |
| 첫 요청만 느림 | P2 또는 안내문구 |
| 오류 문구가 불친절함 | P2 |

## 7. 관리자 보안 점검

- [ ] /admin 주소는 일반 화면에서 노출되지 않음
- [ ] 일반 로그인 화면에 관리자 카드가 보이지 않음
- [ ] 관리자 토큰 없이 MY PAGE 관리자 패널 미노출
- [ ] 관리자 토큰 입력 후 관리자 패널 표시
- [ ] 관리자 모드 해제 후 패널 숨김
- [ ] 공용 PC 사용 후 관리자 모드 해제 안내 표시

운영 기준:

```text
/admin 주소 숨김은 노출 최소화 장치입니다.
실제 보안은 FINPLE_ADMIN_TOKEN 서버 검증으로 유지합니다.
```

## 8. Supabase 데이터 점검

베타 기간에는 Supabase에서 아래 데이터만 주기적으로 확인합니다.

- [ ] users 또는 dev user 데이터 과도 증가 여부
- [ ] portfolios 중복 증가 여부
- [ ] portfolio assets 저장 정상 여부
- [ ] inquiries 문의 접수 정상 여부
- [ ] 종료 처리된 테스트 문의 유지 상태

주의:

```text
운영 중 임의로 DB 데이터를 삭제하기 전에는 반드시 백업 또는 CSV export를 먼저 진행합니다.
```

## 9. 사용자 피드백과 안정성 분류

| 분류 | 기준 | 대응 |
|---|---|---|
| P0 | 접속 불가, 저장 불가, 데이터 유실 | 즉시 수정 |
| P1 | 핵심 기능은 가능하지만 사용 흐름이 막힘 | 우선 수정 |
| P2 | 문구, 디자인, 모바일 가독성 | 다음 개선 |
| HOLD | 결제, 실제 로그인, Pro 기능 | 추후 검토 |
